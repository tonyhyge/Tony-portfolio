## Summary

This phase establishes the quality assurance foundation for the portfolio: Vitest + React Testing Library for unit/component tests, Playwright + axe-core for E2E and accessibility audits, Lighthouse CI for performance enforcement, and dynamic robots.txt/sitemap.xml for SEO. The existing GitHub Actions deploy pipeline gets a test gate before build and an audit gate after deploy. All tooling has been verified against both official documentation and npm registry.

**Primary recommendation:** Install all testing/devDependencies in a single pass, create vitest.config.ts and playwright.config.ts from the exact templates below, then stub the test files before writing the deploy workflow enhancements. This avoids mid-phase dependency resolution failures.

const BASE_URL = "http://localhost:3000"

  testDir: "./tests/e2e",
  fullyParallel: true,

  },
  projects: [
    {
      name: "chromium",
      use: { browserName: "chromium" },
    },
  ],
  webServer: {
    command: "npx serve@latest ./out -l 3000 --no-clipboard",

### Pattern 5: Dynamic Sitemap from Research Data

**What:** Generates sitemap.xml at build time including all sections and research project slugs.

**Source:** [CITED: nextjs.org/docs/app/api-reference/file-conventions/metadata/sitemap]

```typescript
// src/app/sitemap.ts
import type { MetadataRoute } from "next"
import { projects } from "@/app/data/research"

const BASE_URL = "https://tonyhyge.github.io/Tony-portfolio"

export default function sitemap(): MetadataRoute.Sitemap {
  const sectionUrls = [
    {
      url: BASE_URL,
      lastModified: new Date(),
      changeFrequency: "monthly" as const,
      priority: 1,
    },
  ]

  // Add research project pages from data file
  const researchUrls = projects.map((project) => ({
    url: `${BASE_URL}/#research`,
    lastModified: new Date(),
    changeFrequency: "monthly" as const,
    priority: 0.8,
  }))

  return [...sectionUrls, ...researchUrls]
}
```

**Pitfall:** For a single-page portfolio with anchor sections, sitemap entries point to `/#section`. Research slugs are encoded in the single page (filterable), not separate routes. If separate research detail pages are added later, they get their own URLs.

### Anti-Patterns to Avoid

- **Serving dev server for E2E tests:** `next dev` includes HMR and middleware that can mask production bugs. Always serve the static export (`serve ./out`) for E2E and Lighthouse.
- **Using `getByTestId` as default:** Prefer `getByRole`, `getByText`, `getByLabelText` for accessible queries. Fall back to `testid` only for elements without semantic roles.
- **Ignoring `data-slot` attribute:** shadcn components use `data-slot` attributes; the `@testing-library/jest-dom` matchers work with these natively.
- **Running Lighthouse against localhost:3000 dev server:** Lighthouse scores on dev server are unreliable. Always use the production build output.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Test runner | Custom test harness | Vitest | ESM-native, Next.js compatible, built-in coverage, watch mode |
| DOM assertions | Custom matchers | @testing-library/jest-dom | 20+ matchers (toBeInTheDocument, toHaveAttribute, toHaveClass, etc.) |
| E2E browser automation | Puppeteer or raw CDP | Playwright | Cross-browser, auto-wait, trace viewer, CI-optimized |
| Accessibility audit | Manual checklist | @axe-core/playwright | Scans all page content, 100+ WCAG rules, CI-friendly |
| Performance audit | Manual measurement | Lighthouse CI | Standardized scoring, assertion gates, trend tracking |

**Key insight:** Testing and auditing infrastructure has been solved by mature open-source tools with official Next.js compatibility. Every custom alternative is demonstrably worse — more fragile, slower, or less complete.

## Common Pitfalls

### Pitfall 1: Vitest cannot resolve `@/` imports without `vite-tsconfig-paths`
**What goes wrong:** Tests fail with "Cannot find module '@/lib/utils'" even though the app builds.
**Why it happens:** Vitest uses Vite's resolver, not Next.js webpack. It does not read tsconfig paths by default.
**How to avoid:** Always include `vite-tsconfig-paths` in the vitest config plugins array.
**Warning signs:** `ERR_MODULE_NOT_FOUND` for `@/` imports in test output.

### Pitfall 2: Playwright webServer command fails on macOS due to clipboard permission
**What goes wrong:** `serve` command prompts for clipboard access on macOS, causing CI-like local test runs to hang or fail.
**Why it happens:** `serve` v14 defaults to clipboard integration for copyable directory listings.
**How to avoid:** Append `--no-clipboard` flag to the serve command.
**Warning signs:** Interactive prompt "serve wants to access clipboard" during test runs.

### Pitfall 3: Lighthouse CI staticDistDir path mismatch
**What goes wrong:** `lhci autorun` reports "No HTML files found" or scores 0.
**Why it happens:** `staticDistDir` points to the wrong directory or the build step hasn't run.
**How to avoid:** Ensure `npm run build` completes before `lhci autorun`. Set `staticDistDir` to `./out`.
**Warning signs:** Lighthouse report shows 0s for all categories or "no data".

### Pitfall 4: axe-core Playwright test fails because inject/analyze API was called like the standalone version
**What goes wrong:** `AxeBuilder` throws or returns empty results.
**Why it happens:** The `@axe-core/playwright` package uses a different API than `axe-core` CLI. Methods are `withTags()`, `withRules()`, `analyze()` — not `inject()`/`run()`.
**How to avoid:** Use `new AxeBuilder({ page }).withTags([...]).analyze()` pattern. No manual inject needed.
**Warning signs:** `TypeError: page.inject is not a function` or `axe.run is not a function`.

**Deprecated/outdated:**
- Jest for new Next.js projects — Vitest is now the Next.js team's recommendation
- Manual accessibility checklists — axe-core automates WCAG AA conformance checks
- `fireEvent` in Testing Library — prefer `@testing-library/user-event` for realistic interaction simulation
