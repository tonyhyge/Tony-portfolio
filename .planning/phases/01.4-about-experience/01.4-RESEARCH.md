# Phase 1.4: About & Experience - Research

**Researched:** 2026-06-20
**Domain:** About section (narrative bio + stats), experience timeline (connected vertical timeline with dots, content panels, tech tags)
**Confidence:** HIGH

## Summary

This phase builds the About section (narrative bio card + stats block) and Experience section (vertical connected timeline with role entries) on the single-page portfolio home page. Both sections use the existing `Section` component wrapper with different content internals.

The About section is a split layout: narrative paragraph (left) telling the story AI Student -> AI Engineer -> Researcher, alongside a compact stats block (right) with three statistics. No decorative imagery -- text and stats only, per D-04.

The Experience section is a custom-built connected timeline -- not a library component. shadcn/ui has no timeline component. The timeline uses a vertical accent-colored line with dot nodes, and content blocks styled as simple panels (not full `Card` components with header/footer pattern, per D-06). Each entry includes role, company, location, dates, description, and a row of tech tags using the existing shadcn `Badge` component.

The `Experience` data interface in `src/app/data/experience.ts` needs two new fields: `techTags: string[]` (required) and `companyUrl?: string` (optional).

**Primary recommendation:** Build the timeline as a custom `ExperienceTimeline` component using Tailwind utility classes (no library needed). Use `border-l` on entries or pseudo-element for the vertical line, accent-colored dot nodes, and existing `Badge` for tech tags. The About section uses a simple flex/grid split layout with `Section` wrapper.

### Pattern 2: Connected Timeline with CSS Line
**What:** Vertical timeline where the line is created using `border-l` on each entry wrapper. Dots are absolutely-positioned `div` elements on the line. Content panels sit to the right of the line.
**When to use:** The Experience section. This is a custom pattern -- not from a library.
**Implementation approach:**
```tsx
// Pseudocode for timeline structure
// Desktop: dot-on-left, content-on-right
// Mobile: dot-on-left (smaller), content fills remaining space

function ExperienceTimeline({ experiences }) {
  return (
    <div className="relative">
      {experiences.map((exp, i) => (
        <ExperienceEntry key={i} experience={exp} isLast={i === experiences.length - 1} />
      ))}
    </div>
  );
}

function ExperienceEntry({ experience, isLast }) {
  return (
    <div className="relative pl-8 pb-8 last:pb-0">
      {/* Vertical line (connector) */}
      {!isLast && (
        <div className="absolute left-[11px] top-3 bottom-0 w-px bg-border" />
      )}
      {/* Dot node */}
      <div className="absolute left-0 top-[5px] flex items-center justify-center">
        <div className="size-2.5 rounded-full bg-accent ring-4 ring-background" />
      </div>
      {/* Content panel */}
      <div className="space-y-2">
        <div>
          <h3 className="font-heading text-lg font-semibold">{experience.role}</h3>
          <p className="text-sm text-muted-foreground">
            {experience.company} · {experience.location}
          </p>
          <p className="text-xs text-muted-foreground">
            {experience.startDate} — {experience.endDate}
          </p>
        </div>
        <p className="text-sm text-foreground/80">{experience.description}</p>
        {experience.highlights.length > 0 && (
          <ul className="list-disc list-inside text-sm text-muted-foreground space-y-1">
            {experience.highlights.map((h, i) => <li key={i}>{h}</li>)}
          </ul>
        )}
        {experience.techTags.length > 0 && (
          <div className="flex flex-wrap gap-1.5 pt-1">
            {experience.techTags.map((tag) => (
              <Badge key={tag} variant="secondary">{tag}</Badge>
            ))}
          </div>
        )}
      </div>
    </div>
  );
}
```
**Key CSS properties:** `pl-8` (space for the line + dot), `absolute left-[11px]` (line aligned to dot center), `ring-4 ring-background` (dot mask: hides the line behind the dot so it looks like the dot sits ON the line, not at its edge).

## Common Pitfalls

### Pitfall 1: Timeline Dot/Line Alignment
**What goes wrong:** The dot appears offset from the vertical line, or the line extends past the last entry.
**Why it happens:** Inconsistent padding/margin calculations. The `left` position of the line and the `top` position of the dot are calculated independently but must align.
**How to avoid:** Use the same `left` offset for both line and dot. The line width is `w-px` (1px). Center the dot at the same horizontal position: `left-[calc(Xpx)]` where X = the total padding from the container's edge. A proven approach: set dot at `left-0`, dot size `size-2.5` (10px), line at `left-[11px]` (center of 10px dot + 1px offset = 11px). Add `ring-4 ring-background` to the dot so it masks the line behind it.
**Warning signs:** Dot visibly sits to the side of the line, or the line pokes out above/below dot.

### Pitfall 2: Misspelling Timeline Dates
**What goes wrong:** Dates display as raw "YYYY-MM" strings without formatting, or date comparison breaks.
**Why it happens:** The existing data stores dates as `"2024-06"` strings. Without explicit formatting, the raw string appears in the UI.
**How to avoid:** Format dates client-side: `formatDate("2024-06")` -> "June 2024". A simple utility function like `new Date(exp.startDate + "-01").toLocaleDateString("en-US", { year: "numeric", month: "long" })` will convert. Store the raw value in data but always format at render time. For `"Present"`, special-case display.
**Warning signs:** Dates show as "2024-06" instead of "June 2024".

### Pitfall 3: Responsive Timeline Collapse
**What goes wrong:** The desktop timeline layout (dot + line on left, content on right) breaks on mobile, showing awkward spacing or overlapping content.
**Why it happens:** Fixed padding (`pl-8`) designed for desktop may not scale down properly, or the timeline line may overlap text on narrow viewports.
**How to avoid:** Same timeline structure works on mobile with smaller padding. The `pl-8` pattern with smaller dot size on mobile (`size-1.5 sm:size-2.5`) adapts naturally. On very narrow screens (< 375px), consider reducing `pl-8` to `pl-6`. Add `overflow-visible` to prevent content clipping. Test at all breakpoints per NFR3.
**Warning signs:** Text touches the timeline line on mobile, or content has horizontal scroll.

### Pitfall 4: Stats Block Wrapping Awkwardly
**What goes wrong:** The three stats wrap in an ugly way on tablet viewports -- two on one row, one on the next.
**Why it happens:** CSS grid `grid-cols-3` on desktop collapses to `grid-cols-2` or `grid-cols-1` without intentional breakpoint control.
**How to avoid:** Use `flex-wrap` with `justify-center` or a responsive grid: `grid grid-cols-1 sm:grid-cols-3`. Each stat item should have `min-w-0` and allow natural text wrapping within. Alternatively, the split layout (narrative + stats) already stacks on mobile per D-01, so the stats row can be full-width on mobile and a 3-column row on desktop.
**Warning signs:** Stat items are misaligned, or labels wrap to a single letter per line.

### Pitfall 5: Tech Tags Overflowing Container
**What goes wrong:** With many tech tags, the Badge row extends beyond the content panel width or wraps with uneven alignment.
**Why it happens:** Long tag text or many tags without `flex-wrap` cause overflow.
**How to avoid:** Use `flex flex-wrap gap-1.5` on the tag container. Each Badge has `w-fit` (from badgeVariants CSS). No fixed width on badges. This is already handled by the existing Badge component styles. [VERIFIED: badge.tsx `w-fit` + `inline-flex`]
**Warning signs:** Horizontal scroll appears on the experience entry, or tags stack into a single vertical column.
