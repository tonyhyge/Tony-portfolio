## Summary

The challenge is displaying **concurrent professional roles** (AI Engineer at AILOGY + Research Assistant at a university lab) on a portfolio timeline where both positions overlap in time. The current single-track vertical timeline (`experience-timeline.tsx`) renders entries sequentially with a single vertical connector line — it cannot express dual identity or concurrent positions. The recommended approach is to extend the existing `Experience` interface with two optional fields: `track?: "industry" | "research"` (discriminates visual treatment) and `groupId?: string` (groups concurrent entries into a shared time-period row). Entries sharing a `groupId` render side-by-side in a 3-column CSS grid (`track | vertical-line | track`) on desktop. On mobile, the grid collapses to a single column with track badge labels. Entries without `groupId` render unchanged as single-column entries, preserving backward compatibility with the existing data file and the single-vertical-line aesthetic from D-06.

**Primary recommendation:** Add `track` + `groupId` optional fields to the `Experience` interface. Group entries by `groupId` at render time. Use CSS grid (`grid-cols-[1fr_auto_1fr]`) for dual-track groups, with a single center vertical line. On mobile (< 768px), collapse to `flex-col` stack with track badges. No new npm dependencies needed.

## Data Model Options

### Option A: `track` + `groupId` on existing interface (Recommended)

```typescript
export interface Experience {
  role: string;
  company: string;
  location: string;
  startDate: string;
  endDate: string | "Present";
  description: string;
  highlights: string[];
  techTags: string[];
  companyUrl?: string;
  // NEW — concurrent role support
  track?: "industry" | "research";
  groupId?: string;
}
```

**How it works:**
- Entries with the same `groupId` are rendered side-by-side in a CSS grid row within the timeline.
- The `track` field determines the visual treatment (optional track badge label on mobile, optional dot color variant on desktop).
- Entries without `groupId` render as single-column entries — fully backward compatible with existing data.

**Pros:**
- Backward compatible — existing entries without these fields render unchanged.
- Flexible — supports any number of concurrent groups, not limited to two tracks.
- Explicit — `track` is a discriminated union, TypeScript enforces valid values.
- Simple grouping logic — single-pass O(n) scan to build groups.
- Flat array shape preserved — no structural change to `experiences: Experience[]`.

**Cons:**
- Author must manually set matching `groupId` for concurrent entries.
- Optional fields widen the interface — every entry could have `undefined` track/groupId.

**Example data after migration:**

### Option B: Date overlap auto-grouping (no schema change)

```typescript
// No new fields — compute groups from date overlaps at render time
function groupByOverlap(experiences: Experience[]): (Experience | Experience[])[]
```

**How it works:** Algorithm detects entries with overlapping `startDate`/`endDate` ranges and auto-groups them visually.

**Pros:**
- No data model change — zero schema migration.
- Automatic — author doesn't manage group IDs.

**Cons:**
- Complex logic for edge cases (touching ranges, partial overlaps, 3+ concurrent roles).
- Implicit grouping — harder to debug and reason about.
- Cannot distinguish "concurrent by design" from "coincidental date overlap."
- **Not recommended** — the explicit `groupId` approach is simpler and more maintainable for a portfolio with ~5-10 entries.

### Option C: Nested `ExperienceGroup` structure

```typescript
export interface ExperienceGroup {
  period: { startDate: string; endDate: string | "Present" };
  entries: Experience[];
}
export const timelineData: (ExperienceGroup | Experience)[] = [...];
```

**How it works:** Top-level union array contains either single entries or groups of concurrent entries.

**Pros:**
- Clean semantic nesting — groups are explicit containers.

**Cons:**
- Breaks backward compatibility — existing flat `experiences: Experience[]` must be restructured.
- Consumer code must handle union type throughout (`if (Array.isArray(item)) ...`).
- Over-engineering for a portfolio data file with ~5-10 entries.

**Recommendation: Option A.** It preserves the existing flat array shape, is backward compatible, and makes the author's intent explicit via `groupId`.

## Architecture Patterns

### Pattern 1: Grouped Timeline Rendering

```tsx
// Group entries by groupId, return interleaved singles and groups
// sorted reverse-chronologically by start date.
function groupExperiences(data: Experience[]): (Experience | Experience[])[] {
  const groups = new Map<string, Experience[]>();
  const singles: Experience[] = [];

  for (const exp of data) {
    if (exp.groupId) {
      if (!groups.has(exp.groupId)) groups.set(exp.groupId, []);
      groups.get(exp.groupId)!.push(exp);
    } else {
      singles.push(exp);
    }
  }

  const allItems: (Experience | Experience[])[] = [...singles, ...Array.from(groups.values())];
  return allItems.sort((a, b) => {
    const dateA = Array.isArray(a) ? a[0].startDate : a.startDate;
    const dateB = Array.isArray(b) ? b[0].startDate : b.startDate;
    return dateB.localeCompare(dateA);
  });
}
```

### Anti-Patterns to Avoid

- **Separate timelines per track:** Two independent side-by-side timelines create duplicate vertical lines and confuse the chronological flow. A single timeline with grouped rows is cleaner.
- **Date overlap auto-detection:** Computing groups from date ranges is brittle and adds unnecessary complexity. Explicit `groupId` is simpler for portfolio-scale data.
- **Heavy animation library for static content:** The CONTEXT.md defers animations to Phase 4.1. Adding framer-motion now violates YAGNI.
- **Hidden track on mobile:** Do not hide one track on small screens — both identities are equally important. Stack them vertically with clear labeling instead.
- **`order` CSS property for reordering:** Using CSS `order` to swap track positions breaks screen reader reading order. Keep DOM order consistent. [CITED: MDN — "Assistive technologies will announce items in original DOM order"]
- **Client-side media queries for responsive layout:** Using `useMediaQuery` to conditionally render grid vs flex causes hydration mismatch in Next.js static export. Use CSS-only Tailwind prefixes (`md:grid`, `md:flex-col`).

## Common Pitfalls

### Pitfall 1: Grid Row Alignment Drift
**What goes wrong:** When the two tracks in a group have different content heights, the vertical center line doesn't span the full height or gaps appear between the line and dots.
**Why it happens:** CSS grid rows auto-size to content. The center line track needs to stretch to match the tallest track.
**How to avoid:** Use `self-stretch` on the center line element so it spans the full grid row height. If dots need vertical alignment to the line, ensure each track entry has consistent top padding.
**Warning signs:** Visual gap between the center line and one track's dot, or the line stopping short of the taller track's bottom.

### Pitfall 2: Mobile Order Ambiguity
**What goes wrong:** On mobile, the industry entry appears after the research entry (or vice versa) inconsistently across groups.
**Why it happens:** Using CSS `order` or grid `col-start` without considering DOM order.
**How to avoid:** Keep DOM order consistent — always render industry track before research track in JSX. The CSS grid handles the visual desktop layout independently. Screen readers follow DOM order.
**Warning signs:** Screen reader announces entries in wrong chronological order on mobile.

### Pitfall 3: Backward Compatibility Break
**What goes wrong:** Adding `track` and `groupId` as required fields breaks existing entries that don't have them.
**Why it happens:** TypeScript strict mode requires all interface fields.
**How to avoid:** Make both `track` and `groupId` optional (`track?:`, `groupId?:`). Default to `undefined` in render logic — entries without `groupId` render as single-column entries with no grouping wrapper.

### Pitfall 4: Hydration Mismatch from Client-Side Responsive
**What goes wrong:** Different rendering between server (static export) and client causes React hydration warning.
**Why it happens:** Using `window.matchMedia` or `useMediaQuery` to conditionally render grid vs flex layout.
**How to avoid:** Use CSS-only responsive patterns (`flex-col md:grid`). Never gate layout rendering on client-side JavaScript for a static-export Next.js app.

### Pitfall 5: Center Line Dot Misalignment with Dual Tracks
**What goes wrong:** The existing single-track dot alignment (from the current `ExperienceEntry` component) conflicts with the dual-track grid layout where each track has its own dot.
**Why it happens:** The current `ExperienceEntry` has `pl-8` left padding and an absolutely-positioned dot at `left-0`. In a dual-track grid, the center line replaces the left-side line, so the dot positioning needs to change.
**How to avoid:** The `DualTrackGroup` wrapper should provide its own dot positioning context. Each track entry in a dual-track group renders without the left-side dot/line — the center line and track-specific dots are rendered by the group wrapper. Alternatively, keep the existing `ExperienceEntry` unchanged and wrap it with a group-level dot + line overlay.

### DualTrackGroup Component

```tsx
// src/components/experience/dual-track-group.tsx
import { Badge } from "@/components/ui/badge";
import { ExperienceEntry } from "./experience-entry";
import type { Experience } from "@/app/data/experience";

interface DualTrackGroupProps {
  entries: Experience[];
  isLast: boolean;
}

export function DualTrackGroup({ entries, isLast }: DualTrackGroupProps) {
  const industry = entries.find((e) => e.track === "industry");
  const research = entries.find((e) => e.track === "research");

  return (
    <div className="flex flex-col gap-4 md:grid md:grid-cols-[1fr_auto_1fr] md:gap-0 md:items-start">
      {/* Industry track */}
      {industry && (
        <div className="md:col-[1] md:pr-6">
          {/* Track badge — mobile only */}
          <Badge variant="outline" className="mb-2 md:hidden">
            Industry
          </Badge>
          <ExperienceEntry experience={industry} isLast={false} />
        </div>
      )}

      {/* Vertical connector — desktop only */}
      <div
        className="hidden md:block md:col-[2] w-px bg-accent self-stretch mx-2"
        aria-hidden="true"
      />

      {/* Research track */}
      {research && (
        <div className="md:col-[3] md:pl-6">
          {/* Track badge — mobile only */}
          <Badge variant="outline" className="mb-2 md:hidden">
            Research
          </Badge>
          <ExperienceEntry experience={research} isLast={false} />
        </div>
      )}

      {/* Bottom connector line to next entry — hidden for last group */}
      {!isLast && (
        <div className="hidden md:block md:col-[1/-1] mt-0">
          <div className="w-px bg-border mx-auto" style={{ height: "1.5rem" }} />
        </div>
      )}
    </div>
  );
}
```

### Updated ExperienceTimeline with Grouping

```tsx
// src/components/experience/experience-timeline.tsx
import { experiences } from "@/app/data/experience";
import { ExperienceEntry } from "./experience-entry";
import { DualTrackGroup } from "./dual-track-group";

function groupExperiences(data: typeof experiences): (typeof experiences | Experience)[] {
  const groups = new Map<string, typeof experiences>();
  const singles: typeof experiences = [];

  for (const exp of data) {
    if (exp.groupId) {
      if (!groups.has(exp.groupId)) groups.set(exp.groupId, []);
      groups.get(exp.groupId)!.push(exp);
    } else {
      singles.push(exp);
    }
  }

  const allItems: (typeof experiences | Experience)[] = [...singles, ...groups.values()];
  return allItems.sort((a, b) => {
    const dateA = Array.isArray(a) ? a[0].startDate : a.startDate;
    const dateB = Array.isArray(b) ? b[0].startDate : b.startDate;
    return dateB.localeCompare(dateA);
  });
}

export function ExperienceTimeline() {
  const grouped = groupExperiences(experiences);

  return (
    <div className="relative space-y-6">
      {grouped.map((item, i) => {
        const isLast = i === grouped.length - 1;
        if (Array.isArray(item)) {
          return <DualTrackGroup key={item[0].groupId!} entries={item} isLast={isLast} />;
        }
        return (
          <ExperienceEntry
            key={`${item.company}-${item.role}`}
            experience={item}
            isLast={isLast}
          />
        );
      })}
    </div>
  );
}
```

## Mobile Strategy

### Collapse Behavior (breakpoint: `md` = 768px)

| Desktop (>= 768px) | Mobile (< 768px) |
|-------------------|------------------|
| 3-column CSS grid: `1fr | auto | 1fr` | Single `flex-col` stack |
| Center vertical line between tracks | No center line |
| Side-by-side entries | Industry entry first, research entry second |
| No track label needed | Track badge (`"Industry"` / `"Research"`) shown above each entry |
| Each track's dot aligns to inner edge of center line | Each entry keeps its own left-side dot + line (existing single-track pattern) |

### Implementation

The `DualTrackGroup` component uses `flex flex-col gap-4 md:grid md:grid-cols-[1fr_auto_1fr]`:
- On mobile, the flexbox stack renders both tracks vertically. Each track's `ExperienceEntry` renders its own left-side dot and connector line (unchanged from existing behavior). Track badges are visible.
- On desktop, CSS grid takes over. The center line replaces individual track lines. Track badges are hidden.

**Key insight:** The `ExperienceEntry` component does NOT need to know it's in a dual-track group. The `DualTrackGroup` wrapper handles the grid layout and center line. Each entry renders its own dot/line as before. On desktop, the entry's left-side line is visually redundant with the center line — this is acceptable for MVP, or the entry can receive a `hideLine` prop when inside a group.

## Accessibility Considerations

### Screen Reader Order

- Assistive technologies read DOM order, not visual grid order.
- The HTML structure should place entries in chronological reading order (industry first, then research within a group).
- `order` CSS property should NOT be used to reorder — it breaks screen reader flow. [CITED: MDN Web Docs — flexbox accessibility notes]

### ARIA Strategy

- Timeline is NOT a landmark role — it is decorative presentation.
- Use semantic `<ol>` or `<ul>` for the list of entries if treating as a chronological sequence.
- The vertical line and dots are decorative — mark them `aria-hidden="true"`.
- Date ranges should be human-readable text, not just visual positioning.
- Track badges on mobile should be purely decorative (`aria-hidden="true"`) or use `aria-label` on the parent group to announce the grouping context.

### Reduced Motion

If scroll-triggered animations are added later (Phase 4.1):

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

JavaScript detection pattern (for `react-intersection-observer` animations):

```ts
const prefersReducedMotion = window.matchMedia("(prefers-reduced-motion: reduce)");
if (prefersReducedMotion.matches) {
  // Skip animations — render entries statically visible
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Single vertical timeline only | Grouped timeline with track discriminator | Now | Supports concurrent roles |

**Deprecated/outdated:**
- **Table-based timeline layouts:** Layout tables are inaccessible and break responsive design. Use CSS grid.
- **JavaScript-only media queries for responsive timelines:** Causes FOUC and hydration mismatch in SSR/static contexts. Use Tailwind responsive prefixes.
- **Separate timeline pages for industry vs research:** Modern portfolios show both identities on a single page. The dual-track pattern achieves this without separate sections.

## Open Questions

1. **Should track labels be visible on desktop too, or mobile only?**
   - What we know: Mobile needs them because the side-by-side visual collapses.
   - What's unclear: Whether desktop users benefit from track labels when entries are side-by-side.
   - Recommendation: Mobile only for now. The side-by-side grid layout + subtle visual differentiation is sufficient on desktop. Can add desktop labels later if user feedback requests them.

2. **What visual differentiates industry vs research track on desktop?**
   - What we know: Both tracks share the same content panel style (D-06).
   - What's unclear: Whether dots should have different colors per track, or if a subtle background tint is preferred.
   - Recommendation: Use the same accent-colored dot for both tracks (preserves D-06 design). If differentiation is needed later, use `--accent` for industry and `--accent-secondary` for research (both defined in `globals.css`: `--accent: oklch(0.65 0.18 210)`, `--accent-secondary: oklch(0.55 0.15 280)`). [VERIFIED: globals.css]

3. **Should the vertical line split into two separate lines (one per track) on desktop?**
   - What we know: Current design has a single center line (D-06).
   - What's unclear: Whether dual tracks should each have their own line (like a fork) or share a center line.
   - Recommendation: Single center line between the two tracks. This is cleaner and avoids visual clutter. Each track's dot aligns to the inner edge facing the center line.
