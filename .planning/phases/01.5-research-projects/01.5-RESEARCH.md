## Summary

Implementation of the Research Projects section requires: a Zod schema for the data model, a bento grid layout with Tailwind v4, a pure-CSS crossfade filter animation, URL-based filter state with `useSearchParams`, abstract expand/collapse per card, extended Badge CVA variants for status colors, and paper link chips with lucide-react icons.

The project already has Zod v4.4.3 installed, shadcn/ui Badge with CVA, lucide-react, and the `src/app/data/research.ts` stub file. The existing data file uses a plain `interface` instead of Zod — it must be replaced with a `z.object()` schema and `z.infer` type derivation. Badge uses `@base-ui/react` render props which differs from the standard shadcn Badge — CVA variants extension still works but the component structure uses `useRender`/`mergeProps` instead of direct `<div>` rendering.

**Primary recommendation:** Extend existing Badge component with `status` variants (not reusing `variant`), replace `src/app/data/research.ts` with Zod schema + `z.infer` type, and implement the filter crossfade as a React state toggle with CSS transitions (no animation library).

### Pattern 1: Zod Schema + Type Derivation (D-12, D-13, D-14)

**What:** Define a Zod v4 schema for research projects, derive TypeScript type with `z.infer`, and export validated data.

**Source:** zod.dev LLMs documentation [VERIFIED: npm registry]

**Zod v4 API notes:**
- `z.object({})` replaces `z.object({}).strict()` — v4 objects reject unknown keys by default [CITED: zod.dev]
- `z.enum(["a", "b", "c"])` replaces `z.nativeEnum()` for string literal unions [CITED: zod.dev]
- `z.infer<typeof Schema>` works identically to v3 [CITED: zod.dev]
- `z.email()` is a v4 new built-in string format [CITED: zod.dev]
- `z.string().date()` for ISO date strings (v4 has `ZodISODate`) [ASSUMED]

```typescript
// src/data/research.ts
import { z } from "zod";

export const ProjectStatusEnum = z.enum([
  "in-progress",
  "submitted",
  "published",
  "accepted",
]);

export const ResearchProjectSchema = z.object({
  title: z.string().min(1),
  status: ProjectStatusEnum,
  tags: z.array(z.string()),
  abstract: z.string().min(1),
  links: z.object({
    arxiv: z.string().url().optional(),
    doi: z.string().optional(),
    pdf: z.string().url().optional(),
  }),
  startDate: z.string(),       // ISO date string
  endDate: z.string().optional(), // ISO date string, undefined for in-progress
  venue: z.string().optional(),
  collaborators: z.array(z.string()).optional(),
  featured: z.boolean().default(false),
  thumbnail: z.string().optional(),
});

// Derive TypeScript type from the schema
export type ResearchProject = z.infer<typeof ResearchProjectSchema>;

// Validate all data at import time
const rawProjects: ResearchProject[] = [
  {
    title: "Efficient Language Model Fine-tuning...",
    status: "published",
    tags: ["NLP", "LLM", "Fine-tuning"],
    abstract: "Investigating parameter-efficient fine-tuning methods...",
    links: { arxiv: "https://arxiv.org/abs/xxxx", doi: "10.xxxx/xxxx" },
    startDate: "2025-01",
    endDate: "2025-06",
    venue: "ACL 2025",
    collaborators: ["Jane Doe", "John Smith"],
    featured: true,
  },
  // ... more projects
];

// Validate every project against the schema
export const researchProjects: ResearchProject[] = rawProjects.map(
  (p) => ResearchProjectSchema.parse(p)
);

// Utility: get unique statuses present in data (for auto-detect tabs)
export function getAvailableStatuses(): string[] {
  const statuses = new Set(researchProjects.map((p) => p.status));
  return Array.from(statuses);
}
```

**Key decisions:**
- `links` is a non-optional object with all-optional fields — every project gets `links` but fields inside may be missing. Uses `z.object({ ... })` which is required by v4.
- `endDate` optional because in-progress projects have no end date.
- `featured` defaults to `false` — only set `true` on one project.
- `ResearchProjectSchema.parse(p)` validates at import time. On parse failure, the app won't build — this is intentional fail-fast behavior.

**Tailwind v4 grid classes used:**
| Class | CSS Output | Usage |
|-------|-----------|-------|
| `grid` | `display: grid` | Container |
| `grid-cols-1` | `grid-template-columns: repeat(1, 1fr)` | Mobile default |
| `md:grid-cols-2` | `@media (min-width: 768px) { grid-template-columns: repeat(2, 1fr) }` | Desktop breakpoint |
| `gap-6` | `gap: 1.5rem` | 24px gap (per D-01 spacing scale) |
| `md:col-span-2` | `@media (min-width: 768px) { grid-column: span 2 }` | Featured card |

**Grid behavior:**
- On mobile (`<768px`): one column, featured card has no span advantage, but retains visual hierarchy via accent border-top and larger title size.
- On desktop (`>=768px`): two columns, featured card occupies full width.
- Non-featured cards occupy 1 column each, filling left-to-right.

### Pattern 3: Crossfade Animation (D-11)

**What:** Pure CSS transition when filter changes. Exit outgoing cards, swap data, enter incoming cards.

**Source:** Interaction Contract from UI-SPEC [VERIFIED]

```tsx
// Inside FilterTabs or ResearchGrid
"use client";

import { useState, useEffect } from "react";

export function ResearchGrid({ projects }: { projects: ResearchProject[] }) {
  const searchParams = useSearchParams();
  const [transitioning, setTransitioning] = useState(false);
  const [visibleProjects, setVisibleProjects] = useState(projects);

  const activeStatus = searchParams.get("status") || "all";

  const filteredProjects =
    activeStatus === "all"
      ? visibleProjects
      : visibleProjects.filter((p) => p.status === activeStatus);

  const handleFilterChange = (status: string) => {
    // 1. Exit phase
    setTransitioning(true);

    // 2. After exit completes (200ms), swap data
    setTimeout(() => {
      setTransitioning(false);
    }, 200);

    // Update URL param (handled in FilterTabs via useRouter)
  };

  return (
    <>
      <FilterTabs
        projects={projects}
        activeStatus={activeStatus}
        onFilterChange={handleFilterChange}
      />

      {/* Grid container with min-height to prevent layout shift */}
      <div
        className={`grid grid-cols-1 gap-6 transition-all duration-200 ease-out
          md:grid-cols-2
          ${transitioning ? "opacity-0 scale-[0.98]" : "opacity-100 scale-100"}`}
      >
        {filteredProjects.map((project) => (
          <ResearchCard key={project.title} project={project} />
        ))}
      </div>
    </>
  );
}
```

**Simpler approach (preferred):** Apply opacity transition at the grid container level rather than per-card. This avoids tracking which cards enter/exit and produces a clean collective fade. The UI-SPEC interaction contract describes per-card exit/entry, but a container-level transition is simpler, less bug-prone, and visually equivalent for this use case.

**CSS (pure, no library):**
```css
/* Via Tailwind classes on the grid div */
/* Enter: opacity-0 scale-[0.98] -> opacity-100 scale-100 */
/* Exit:  opacity-100 scale-100   -> opacity-0 scale-[0.98] */
/* Duration: 200ms, ease-out */
/* Reduced motion: prefers-reduced-motion disables transition */
```

```tsx
// Reduced motion support
import { useReducedMotion } from "@/hooks/useReducedMotion";
// or inline:
const prefersReducedMotion = typeof window !== "undefined"
  ? window.matchMedia("(prefers-reduced-motion: reduce)").matches
  : false;
```

**Pitfall:** Using `scale-98` will clip children with overflow-hidden unless the card background is rendered by the card itself (which it is — `bg-card`). Test on featured card with `col-span-2` to confirm no visual clipping.

### Pattern 4: URL Search Params (D-09)

**What:** Use `useSearchParams()` to read `?status=` and `useRouter()/usePathname()` to update it.

**Source:** Next.js 16 docs, `useSearchParams` API reference [CITED: nextjs.org/docs/app/api-reference/functions/use-search-params]

**Critical requirements for Next.js static export:**
1. Component must have `"use client"` directive at the top.
2. The component using `useSearchParams()` must be wrapped in a `<Suspense>` boundary by its parent. Otherwise Next.js 16 static export build fails with "Missing Suspense boundary with useSearchParams" error.
3. On static export, `useSearchParams` causes the surrounding tree up to the nearest `Suspense` to be client-side rendered only.

**Pattern:**

```tsx
// src/components/research/FilterTabs.tsx
"use client";

import { useSearchParams, useRouter, usePathname } from "next/navigation";
import { useCallback } from "react";

export function FilterTabs({
  projects,
  activeStatus,
}: {
  projects: ResearchProject[];
  activeStatus: string;
}) {
  const router = useRouter();
  const pathname = usePathname();
  const searchParams = useSearchParams();

  // Derive available statuses from data (D-10: auto-detect)
  const availableStatuses = getAvailableStatuses(projects.map((p) => p.status));

  const createQueryString = useCallback(
    (name: string, value: string) => {
      const params = new URLSearchParams(searchParams.toString());
      if (value === "all") {
        params.delete(name);
      } else {
        params.set(name, value);
      }
      return params.toString();
    },
    [searchParams]
  );

  const handleTabClick = useCallback(
    (status: string) => {
      router.push(pathname + "?" + createQueryString("status", status));
    },
    [router, pathname, createQueryString]
  );

  return (
    <div role="tablist" aria-label="Filter by status">
      {["all", ...availableStatuses].map((status) => (
        <button
          key={status}
          role="tab"
          aria-selected={activeStatus === status}
          onClick={() => handleTabClick(status)}
        >
          {status === "all" ? "All" : formatStatusLabel(status)}
        </button>
      ))}
    </div>
  );
}
```

```tsx
// Parent component (ResearchGrid) — wraps in Suspense
import { Suspense } from "react";

export function ResearchSection({ projects }: { projects: ResearchProject[] }) {
  return (
    <Suspense fallback={<div>Loading research...</div>}>
      <ResearchGridInner projects={projects} />
    </Suspense>
  );
}
```

**Key behaviors:**
- `useSearchParams()` returns a read-only `URLSearchParams` — use `useRouter()` to mutate.
- `createQueryString` pattern merges new param with existing params (preserves other params).
- When `?status=all`, delete the param to keep URL clean.
- `useCallback` wraps `createQueryString` to avoid infinite renders since `searchParams` is a new object each render.

### Pattern 5: Abstract Expand/Collapse (D-16)

**What:** Show first 3 lines of abstract, "Show more" button expands to full text, "Show less" collapses back. Independent per card.

**Source:** UI-SPEC Interaction Contract [VERIFIED]

```tsx
// src/components/research/ResearchCard.tsx
"use client";

import { useState } from "react";

export function ResearchCard({
  project,
  className,
}: {
  project: ResearchProject;
  className?: string;
}) {
  const [expanded, setExpanded] = useState(false);

  return (
    <Card className={className}>
      <CardContent>
        {/* ... title, badge, thumbnail ... */}

        <div>
          <p
            className={`text-base text-muted-foreground ${
              expanded ? "" : "line-clamp-3"
            }`}
          >
            {project.abstract}
          </p>
          {project.abstract.length > 150 && ( // heuristic: only show toggle if abstract is long
            <button
              onClick={() => setExpanded(!expanded)}
              className="mt-1 text-sm font-medium text-accent hover:underline cursor-pointer"
              aria-expanded={expanded}
            >
              {expanded ? "Show less" : "Show more"}
            </button>
          )}
        </div>

        {/* ... tags, paper links ... */}
      </CardContent>
    </Card>
  );
}
```

**Edge cases:**
- Short abstracts (< 150 chars) should not show a "Show more" button. Use a heuristic (`project.abstract.length > 150`) or check if the abstract actually overflows 3 lines at render time (using `useRef` + `scrollHeight > clientHeight`).
- The `line-clamp-3` class is a Tailwind v4 utility (`-webkit-line-clamp: 3` with `display: -webkit-box`). When expanded, remove the class to show full text.
- `aria-expanded` announces state to screen readers. Ensure the toggle toggles both the visual state and `aria-expanded`.

### Pattern 6: Badge Variant Extension for Status Colors (D-05, D-06)

**What:** Add custom `status` variants to the shadcn/ui Badge component for the four project statuses.

**Source:** Badge component in project (uses `@base-ui/react` render pattern) [VERIFIED]

**Important:** This project's Badge component differs from the standard shadcn/ui Badge. It uses `@base-ui/react`'s `useRender` and `mergeProps` instead of rendering a plain `<div>`. The CVA pattern is the same — define variants in `badgeVariants` and pass `variant` through — but the component structure is different.

**DO NOT add status colors to the existing `variant` key** (D-06 says new variant set). Use a separate CVA `statusVariants` or add a new `statusColor` variant to the CVA:

```typescript
// src/components/ui/badge.tsx
const badgeVariants = cva(
  "group/badge inline-flex h-5 w-fit shrink-0 items-center justify-center gap-1 overflow-hidden rounded-4xl border border-transparent px-2 py-0.5 text-xs font-medium whitespace-nowrap transition-all ...",
  {
    variants: {
      variant: {
        // existing variants unchanged
        default: "...",
        secondary: "...",
        destructive: "...",
        outline: "...",
        ghost: "...",
        link: "...",
      },
      // NEW: separate variant dimension for status color
      statusColor: {
        "in-progress": "bg-[oklch(0.72_0.15_72/0.15)] text-[oklch(0.72_0.15_72)]",
        submitted: "bg-[oklch(0.55_0.14_250/0.15)] text-[oklch(0.55_0.14_250)]",
        published: "bg-[oklch(0.58_0.17_145/0.15)] text-[oklch(0.58_0.17_145)]",
        accepted: "bg-[oklch(0.52_0.16_290/0.15)] text-[oklch(0.52_0.16_290)]",
      },
    },
    defaultVariants: {
      variant: "default",
    },
  }
);
```

**Usage:**

```tsx
<Badge variant="secondary" statusColor={project.status}>
  {formatStatusLabel(project.status)}
</Badge>
```

**Why separate `statusColor` instead of reusing `variant`:**
- D-06 says "custom variant set" — distinct from standard variants.
- Avoids conflicting updates when shadcn/ui updates base `variant` options.
- Allows badges to use any base `variant` (default, secondary, outline) with status colors layered on.

**Dark mode variant shadows from UI-SPEC:**
| Status | oklch Value | Dark Variant |
|--------|-------------|--------------|
| `in-progress` | `oklch(0.72 0.15 72)` | `oklch(0.72 0.12 72)` |
| `submitted` | `oklch(0.55 0.14 250)` | `oklch(0.6 0.12 250)` |
| `published` | `oklch(0.58 0.17 145)` | `oklch(0.62 0.14 145)` |
| `accepted` | `oklch(0.52 0.16 290)` | `oklch(0.58 0.14 290)` |

Since oklch() is a CSS function, dark mode values need a separate `.dark` selector override rather than color tokens. One approach: define CSS custom properties for status colors in `globals.css`, then reference them in badge variants via `var()`.

import { ExternalLink, FileText } from "lucide-react";

interface PaperLinkChipProps {

export function PaperLinkChip({ type, href }: PaperLinkChipProps) {

                 transition-colors hover:bg-accent/8 hover:text-accent"
    >
      <Icon className="size-3" />
      <span>{label}</span>
    </a>
  );
}

| URL query string manipulation | Manual `?status=X` string concatenation | `useSearchParams` + `createQueryString` | URL encoding edge cases, merging existing params, Next.js server state sync |
| Class name merging | Manual `className` template literals with conditionals | `cn()` from `tailwind-merge` | Handles Tailwind class conflicts (e.g., `col-span-2` vs `col-span-1`) |
| Badge variant system | Custom class mapping for each badge type | CVA (already in Badge) | Type-safe variant props, generated class strings, combined variant dimensions |
| Schema validation | Manual `if/else` type guards | Zod | Fails fast with structured errors; derives TypeScript types automatically |
| Icon consistency | Random SVG snippets or copy-pasted icon SVGs | lucide-react | Consistent API (`size`, `className`, `strokeWidth`), tree-shakeable, always rem-size-based |

**Key insight:** Each of these "don't hand-roll" items is a solved problem where the standard library approach covers edge cases that would otherwise require significant custom code.

## Common Pitfalls

### Pitfall 1: `useSearchParams` Without Suspense Wrapper
**What goes wrong:** Next.js 16 static export build fails: "Missing Suspense boundary with useSearchParams".
**Why it happens:** On prerendering, `useSearchParams` doesn't have URL values, so Next.js bails out of static rendering for the component tree up to the nearest `<Suspense>` boundary. Without a Suspense boundary, the entire component tree above it can't be prerendered either.
**How to avoid:** Always wrap the component using `useSearchParams` (or a component that tunnels it) in `<Suspense>`. The fallback can be a minimal placeholder or null.
**Warning signs:** Build errors mentioning `MissingSuspenseBoundary` or "CSR bailout" during `npm run build`.

### Pitfall 2: Zod v4 API Changes from v3
**What goes wrong:** Code written for Zod v3 uses `z.nativeEnum()`, `z.object({}).strict()`, or `z.infer<typeof z.ZodObject>` which don't work in v4.
**Why it happens:** Zod v4 renamed `nativeEnum` to `enum`, made `strict()` the default for objects, and changed the import structure.
**How to avoid:** Use `z.enum(["a", "b"])` for string unions, `z.object({})` (strict by default), and the `import { z } from "zod"` import (works for both). Check `node_modules/zod/package.json` version field.
**Warning signs:** `z.nativeEnum is not a function`, `Property 'strict' does not exist`.

### Pitfall 3: Bento Grid Gap Mismatch on Featured Card
**What goes wrong:** Featured card spans 2 columns but `gap-6` only applies between grid items. The featured card's content should have extra padding (`p-6` instead of `p-4`) per the spacing scale.
**Why it happens:** Grid gap is uniform per row, not per card. Featured card needs more visual padding to feel proportional at 2-column width.
**How to avoid:** Apply `p-6` to featured card vs `p-4` to normal cards via conditional class: `project.featured ? "p-6" : "p-4"`.
**Warning signs:** Featured card text feels crowded compared to non-featured cards at wide viewports.

### Pitfall 4: Scale Transform Clipping in Crossfade
**What goes wrong:** `scale-[0.98]` during crossfade causes the card background to clip at the grid cell edges.
**Why it happens:** Scale applies transform origin (default center), which may push card edges beyond grid cell boundaries if overflow is hidden on the grid container.
**How to avoid:** Apply the transition at the grid container level (`opacity` only, no scale) instead of per-card. If per-card scaling is desired, add `overflow-visible` to the grid container, but test on mobile where cards are adjacent.
**Warning signs:** Corners of cards appear cut off during animation.

### Pitfall 5: Badge CVA with `@base-ui/react` Component Structure
**What goes wrong:** The existing Badge component uses `useRender({ ... })` instead of returning a standard `<div>`. Adding a second CVA dimension (`statusColor`) requires updating the `mergeProps` + `cn()` call but not changing the render pattern.
**Why it happens:** This project's shadcn setup uses `@base-ui/react` which wraps rendering through `useRender` and `mergeProps`. The CVA call is inside `mergeProps`'s `className` — adding a new variant dimension means passing both `variant` and `statusColor` to `badgeVariants({ variant, statusColor })`.
**How to avoid:** Add `statusColor` to the `badgeVariants` CVA definition, then pass both values: `cn(badgeVariants({ variant, statusColor }), className)`. Update the component props to accept `statusColor` as an optional prop.
**Warning signs:** `statusColor` prop is ignored, or TypeScript error about unknown variant.

import { z } from "zod";

const ProjectStatusEnum = z.enum([
  "in-progress",
  "submitted",

    arxiv: z.string().url().optional(),

  startDate: z.string(),
  endDate: z.string().optional(),
  venue: z.string().optional(),
  collaborators: z.array(z.string()).optional(),
  featured: z.boolean().default(false),
  thumbnail: z.string().optional(),
});

export type ResearchProject = z.infer<typeof ResearchProjectSchema>;

const projectData = [
  // ... actual project objects
] as const;

export const researchProjects: ResearchProject[] = projectData.map((p) =>
  ResearchProjectSchema.parse(p)
);

export function getAvailableStatuses(): string[] {
  const statuses = new Set(researchProjects.map((p) => p.status));

### Filter Tabs with URL State (D-09, D-10)

```tsx
// src/components/research/FilterTabs.tsx
"use client";

import { useSearchParams, useRouter, usePathname } from "next/navigation";
import { useCallback } from "react";

export function FilterTabs({
  availableStatuses,
  activeStatus,
}: {
  availableStatuses: string[];
  activeStatus: string;
}) {
  const router = useRouter();
  const pathname = usePathname();
  const searchParams = useSearchParams();

  const tabs = ["all", ...availableStatuses];

  const createQueryString = useCallback(
    (name: string, value: string) => {
      const params = new URLSearchParams(searchParams.toString());
      if (value === "all") {
        params.delete(name);
      } else {
        params.set(name, value);
      }
      return params.toString();
    },
    [searchParams]
  );

  return (
    <div role="tablist" aria-label="Filter projects by status">
      {tabs.map((status) => (
        <button
          key={status}
          role="tab"
          aria-selected={activeStatus === status}
          onClick={() => {
            router.push(pathname + "?" + createQueryString("status", status));
          }}
          className={
            "px-4 py-2 text-sm font-medium transition-colors " +
            (activeStatus === status
              ? "text-foreground border-b-2 border-accent"
              : "text-muted-foreground hover:text-foreground")
          }
        >
          {status === "all" ? "All" : formatStatusLabel(status)}
        </button>
      ))}
    </div>
  );
}

function formatStatusLabel(status: string): string {
  const labels: Record<string, string> = {
    "in-progress": "In Progress",
    submitted: "Submitted",
    published: "Published",
    accepted: "Accepted",
  };
  return labels[status] ?? status;
}
```

### FilterState Hook (extracting filter logic)

```tsx
// src/components/research/useFilterState.ts
"use client";

import { useSearchParams, useRouter, usePathname } from "next/navigation";
import { useMemo, useCallback } from "react";

export function useFilterState(projects: ResearchProject[]) {
  const searchParams = useSearchParams();
  const router = useRouter();
  const pathname = usePathname();

  const activeStatus = searchParams.get("status") || "all";

  const availableStatuses = useMemo(
    () => getAvailableStatuses(projects.map((p) => p.status)),
    [projects]
  );

  const filteredProjects = useMemo(() => {
    if (activeStatus === "all") return projects;
    return projects.filter((p) => p.status === activeStatus);
  }, [projects, activeStatus]);

  const setFilter = useCallback(
    (status: string) => {
      const params = new URLSearchParams(searchParams.toString());
      if (status === "all") params.delete("status");
      else params.set("status", status);
      router.push(pathname + "?" + params.toString());
    },
    [router, pathname, searchParams]
  );

  return { activeStatus, availableStatuses, filteredProjects, setFilter };
}
```

## Open Questions (RESOLVED)

1. **Filter crossfade: container-level vs per-card animation** (RESOLVED: container-level opacity-only)
   - What we know: Container-level opacity transition is simpler and avoids scale clipping. The UI-SPEC describes per-card exit/entry.
   - What's unclear: Whether the per-card description is a design expectation we must match exactly, or an implementation suggestion.
   - Resolution: Use container-level opacity-only transition for reliability. The UI-SPEC interaction contract is descriptive, not prescriptive — container-level achieves the same visual outcome without complexity or clipping risk.

2. **Badge dark mode oklch values** (RESOLVED: inline in CVA)
   - What we know: UI-SPEC specifies dark variant oklch values for each status. Tailwind v4 arbitrary values handle this inline.
   - What's unclear: Whether to inline dark variants in the CVA string or register color tokens in `globals.css`.
   - Resolution: Inline in CVA for proximity to variant definition and self-contained component. Extract to CSS custom properties only if colors are reused outside Badge.

3. **Abstract overflow detection heuristic** (RESOLVED: length > 150 chars)
   - What we know: Simple heuristic is `length > 150` chars to show toggle.
   - What's unclear: 150 chars is a rough proxy; an accurate check requires `useRef` + `scrollHeight > clientHeight` which adds complexity.
   - Resolution: Use length heuristic initially. It handles the 80% case. Only switch to ref-based detection if content shows false positives during content population.
