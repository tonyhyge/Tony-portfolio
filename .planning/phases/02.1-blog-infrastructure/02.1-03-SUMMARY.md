# Phase 2.1 Plan 03: Dynamic Blog Route Summary

Created the `/blog/[slug]` dynamic route that renders MDX blog posts with syntax highlighting at build time.

## Key Decisions

- **RSC MDXRemote** (`next-mdx-remote/rsc`) instead of `serialize` + `MDXRemote` from the main module — the RSC variant accepts raw MDX source and options directly, eliminating separate serialization and enabling zero runtime JS.
- **Async params** — Next.js 16 passes page `params` as `Promise<{ slug: string }>`, consistent with the Next.js 15+ async params model.
- **Dual syntax theme** — rehype-pretty-code configured with `github-dark-dimmed` (lower contrast for dark bg) and `github-light` (clean light theme), with `keepBackground: false` to respect the site's own background colors.

## Created

### `src/app/blog/[slug]/page.tsx`
- `generateStaticParams()` reads all slugs from `content/blog/` at build time
- Async Server Component with typed `Promise<{ slug: string }>` params
- MDXRemote from `next-mdx-remote/rsc` renders content via rehype-pretty-code
- Post metadata header: title, formatted date, reading time, tags
- Typography via `prose prose-neutral dark:prose-invert` classes
- Accessible `lang="en"` on article element

## Deviations from Plan

### Rule 2 — Missing critical functionality
- `getPostBySlug()` parsed frontmatter and content via gray-matter, but only returned metadata — the MDX `content` string was discarded. Route needs content to serialize. Added `content` field to `BlogPostMeta` type and returned it from the function.

### Rule 3 — Blocking issues
- **Params typing**: Plan stated "Next.js 16 uses unwrapped params (no async/Promise needed)". This is incorrect — Next.js 16 uses async `Promise<{ slug: string }>` params consistent with the model introduced in Next.js 15. Build failed with `undefined.mdx` error before fix.
- **Missing typography plugin**: `prose` classes depend on `@tailwindcss/typography`. Installed and registered in globals.css.
- **Wrong MDXRemote import**: Plan specified `next-mdx-remote` (client module). For Server Components, `next-mdx-remote/rsc` is the correct import — it accepts `source: VFileCompatible` + `options` directly instead of a pre-serialized result.
