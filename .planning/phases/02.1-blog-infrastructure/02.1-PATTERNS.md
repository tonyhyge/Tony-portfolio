```typescript
export const BlogPostSchema = z.object({
  title: z.string().min(1),
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  tags: z.array(z.string()).default([]),
  excerpt: z.string().optional(),
  cover: z.string().optional(),
  draft: z.boolean().default(false),
  updated: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
  series: z.string().optional(),
  canonical: z.string().url().optional(),
})

export type BlogPost = z.infer<typeof BlogPostSchema>

export type BlogPostMeta = BlogPost & { slug: string; readingTime: number }
```

**Data loading pattern** (lines 116-130 in research.ts) — use `readdirSync` / `readFileSync` at build time:
```typescript
import { readFileSync, readdirSync } from "fs"
import { join } from "path"
import matter from "gray-matter"

const POSTS_DIR = join(process.cwd(), "content", "blog")

export function getAllSlugs(): string[] {
  return readdirSync(POSTS_DIR)
    .filter((f) => f.endsWith(".mdx"))
    .map((f) => f.replace(/\.mdx$/, ""))
}

export function getPostBySlug(slug: string): BlogPostMeta {
  const source = readFileSync(join(POSTS_DIR, `${slug}.mdx`), "utf-8")
  const { data, content } = matter(source)
  const parsed = BlogPostSchema.parse(data)
  const wordCount = content.split(/\s+/).length
  const readingTime = Math.ceil(wordCount / 200)
  return { ...parsed, slug, readingTime }
}

export function getAllPosts(): BlogPostMeta[] {
  return getAllSlugs()
    .map((slug) => getPostBySlug(slug))
    .filter((post) => !post.draft)
    .sort((a, b) => new Date(b.date).getTime() - new Date(a.date).getTime())
}
```

**Derived helpers pattern** (lines 132-135 in research.ts) — same pattern:
```typescript
export function getAllTags(): string[] {
  const tags = new Set(allPosts.flatMap((p) => p.tags))
  return Array.from(tags).sort()
}
```

**Key differences from analog:**
- Uses `gray-matter` instead of hardcoded data array
- Uses `fs` module (Node runtime at build time only)
- `slug` comes from filename, not derived from title
- `readingTime` is computed, not stored in frontmatter

### Zod + TypeScript Schema Pattern

**Source:** `src/app/data/research.ts` lines 1-31
**Apply to:** `src/app/data/blog.ts`

```typescript
import { z } from "zod"

export const MySchema = z.object({
  // field: z.type().constraint().default()
})

export type MyType = z.infer<typeof MySchema>

// At the boundary:
const parsed = MySchema.parse(rawData)  // throws on invalid
const safe = MySchema.safeParse(rawData) // returns { success, data/error }
```

Follow this exact convention:
1. Export schema first (`PascalCase + Schema`)
2. Re-export inferred type (`z.infer<typeof Schema>`)
3. Use `.parse()` at the boundary to throw on invalid data
4. Use `.default()` for optional fields with fallback values

## Gotchas

1. **`process.cwd()` in data modules:** `getAllPosts()` uses `process.cwd()` to resolve `content/` directory at build time. This works under `next build` but may return different values depending on the calling context. Always join with `process.cwd()` explicitly.

2. **`fs` calls only at module top-level or in exported functions called by `generateStaticParams()`:** Do not call `readdirSync` inside a React component or `useEffect`. All file I/O happens at build time during `generateStaticParams()` or during data module initialization.

3. **Zod v4 vs v3 syntax:** The project uses Zod v4 (`^4.4.3` in `package.json` line 30). The `z.infer<>` and `z.object()` APIs are the same, but check for any breaking changes in Zod v4 before writing the schema.

4. **`package.json` will need new dependencies:** `next-mdx-remote`, `gray-matter`, `rehype-pretty-code`, `shiki`, `@types/gray-matter` must be `npm install`-ed. The pattern (from existing deps) is to install and let `package-lock.json` resolve versions.

5. **`basePath` in production links:** The static export deploys to `/Tony-portfolio/`. Hardcoded links (`/blog/my-post`) will 404 in production. Use relative links or prepend the base path conditionally as done in `not-found.tsx` line 25.

6. **No React embedding in Phase 2.1:** The `MDXRemote` component supports embedded React components, but the CONTEXT.md explicitly scopes this to "backend/infrastructure." Custom MDX component mapping (e.g., `<Image>`, `<Tweet>`) belongs in Phase 2.2 or Phase 2.3.

7. **`generateStaticParams()` return shape:** For `app/blog/[slug]/page.tsx`, the function must return `{ slug: string }[]` objects. The route component receives `{ params: { slug: string } }`.

8. **Draft post handling:** DRAFT posts should be filtered from `getAllPosts()` but still accessible via direct URL if `getPostBySlug()` is called directly. The `generateStaticParams()` function — decide whether to include drafts or not. If excluded, drafts won't generate at build time and will 404. If included, they should be hidden from list but accessible by URL.
