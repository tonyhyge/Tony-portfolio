## Summary

Reviewed the hero section implementation (6 files). The component structure is clean with good separation of concerns — Hero, HeroCtas, HexagonBadge, and ScrollIndicator are properly factored. Two critical issues were found: a dead/broken "Download CV" link and an invalid `nativeButton` prop on the Button component. Three warnings cover accessibility and SVG filter scoping concerns.

## Critical Issues

### CR-01: "Download CV" link has empty href and opens current page in new tab

**File:** `src/components/hero/HeroCtas.tsx:26-28`
**Issue:** `href="#"` combined with `target="_blank"` opens the current page in a new tab instead of navigating to a CV/resume file. The `#` fragment resolves to the current URL. This is a broken link — the CV file likely does not exist at any path.

**Fix:**
```tsx
<a
  href="/Tony-Ge-Huy-CV.pdf"  // or wherever the CV file lives in public/
  target="_blank"
  rel="noopener noreferrer"
  // ...
>
```

Place the actual CV PDF in `public/` and reference it with an absolute path.

### CR-02: `nativeButton={false}` is not a valid prop on base-ui Button

**File:** `src/components/hero/HeroCtas.tsx:12` and `src/components/hero/HeroCtas.tsx:20`
**Issue:** `ButtonPrimitive.Props` from `@base-ui/react/button` does not include a `nativeButton` prop. This prop is silently ignored (spread into the DOM as a custom attribute), having no effect. The `render={<a .../>}` pattern alone controls whether a native `<button>` is rendered. Passing `nativeButton={false}` gives the false impression of controlling button behavior.

**Fix:** Remove the `nativeButton` prop entirely. The `render` prop is sufficient to override the underlying element:
```tsx
<Button
  variant="default"
  size="lg"
  render={<a href="#experience" />}
>
  View Projects
</Button>
```

## Warnings

### WR-01: HexagonBadge `alt` defaults to empty string, hiding images from screen readers

**File:** `src/components/hero/HexagonBadge.tsx:7,13-14`
**Issue:** `alt = ""` means even when `src` is provided (making the component render an `<img>`), the alt text defaults to empty — the image is invisible to screen reader users. The JSDoc comment says "required when src is provided" but this is not enforced by the type system.

**Fix:** Make `alt` required when `src` is provided. Either split into two component variants (decorative vs. image) or enforce at the type level:
```tsx
interface HexagonBadgeWithImage {
  src: string;
  alt: string;
  className?: string;
}

interface HexagonBadgeDecorative {
  src?: never;
  alt?: never;
  className?: string;
}

type HexagonBadgeProps = HexagonBadgeWithImage | HexagonBadgeDecorative;
```

Or at minimum, add a runtime warning:
```tsx
if (src && !alt) {
  console.warn("HexagonBadge: alt is required when src is provided");
}
```

### WR-02: SVG filter IDs are global — multiple HexagonBadge instances will collide

**File:** `src/components/hero/HexagonBadge.tsx:54,61`
**Issue:** `<filter id="hex-glow">` and `<linearGradient id="hex-stroke">` use static IDs. If multiple `HexagonBadge` instances render on the same page (e.g., in a projects section later), the SVG filter IDs collide. All badges will reference the first rendered filter/gradient definition.

**Fix:** Generate unique IDs per instance:
```tsx
const id = React.useId();
// Then use: id={`hex-glow-${id}`} and `url(#hex-glow-${id})`
```

### WR-03: ScrollIndicator creates multiple rAF frames without throttling

**File:** `src/components/hero/ScrollIndicator.tsx:13-18`
**Issue:** Every scroll event fires a new `requestAnimationFrame` call. If the user scrolls rapidly, multiple rAF callbacks queue up. The cleanup only cancels the last `frameId`, leaving earlier callbacks to fire after unmount. Not a memory leak (rAF callbacks are one-shot), but can cause unnecessary state updates.

**Fix:** Throttle with a flag to skip rAF if one is already pending:
```tsx
let frameId: number | null = null;

const handleScroll = () => {
  if (frameId !== null) return;
  frameId = requestAnimationFrame(() => {
    const scrollY = window.scrollY;
    const fadeRange = 200;
    setOpacity(Math.max(0, 1 - scrollY / fadeRange));
    frameId = null;
  });
};
```

### IN-01: Unused CSS animation defined but never consumed

**File:** `src/app/globals.css:50-56`
**Issue:** `--animate-gradient-shift` and its `@keyframes gradient-shift` are defined in `@theme` but not used by any of the reviewed components. Dead code unless consumed by components outside the review scope.

**Fix:** If this animation will be used by future phases, leave as-is. Otherwise remove it. Verify usage across the full codebase.

### IN-02: `HexagonBadge` duplicate clip-path polygon values

**File:** `src/components/hero/HexagonBadge.tsx:31,41`
**Issue:** The hexagon `clipPath: "polygon(50% 0%, 100% 25%, 100% 75%, 50% 100%, 0% 75%, 0% 25%)"` is duplicated in the outer container (line 31) and the inner `<img>` (line 41). DRY violation — if the shape changes, both must be updated.

**Fix:** Extract to a constant:
```tsx
const HEXAGON_CLIP = "polygon(50% 0%, 100% 25%, 100% 75%, 50% 100%, 0% 75%, 0% 25%)";
```

### IN-03: Hero component missing reduced-motion consideration for ScrollIndicator

**File:** `src/components/hero/Hero.tsx:34,44`
**Issue:** ScrollIndicator renders `animate-bounce-subtle` which has a `motion-reduce:animate-none` fallback (good). However, on desktop it is positioned absolutely at the bottom of a `90vh` section — users with vestibular disorders who avoid scrolling may never see this indicator. Consider whether the scroll indicator provides value or should be replaced with a visible section divider or "next section" arrow.

**Fix:** This is a UX consideration rather than a code issue. The current implementation handles motion-reduce correctly at the component level.
